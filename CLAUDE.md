# CLAUDE.md

이 문서는 이 저장소에서 AI 코딩 에이전트가 따라야 할 작업 규칙이다.

목표는 불필요한 변경, 과한 추상화, 모호한 추측으로 생기는 재작업을 줄이는 것이다. 이 규칙은 속도보다 신중함을 우선한다. 다만 아주 단순한 작업에서는 절차를 과하게 늘리지 말고, 필요한 만큼만 적용한다.

## 1. 코딩 전에 먼저 생각하기

추측하지 않는다. 모호함을 숨기지 않는다. 트레이드오프를 드러낸다.

구현 전에 다음을 지킨다.

- 가정을 명확히 말한다. 확실하지 않으면 질문한다.
- 요구사항 해석이 여러 개라면 가능한 해석을 먼저 제시한다.
- 여러 선택지가 있다면 장단점을 말한다. 조용히 하나를 고르지 않는다.
- 더 단순한 접근이 있다면 먼저 말한다.
- 요구사항이나 접근 방식이 과하다고 판단되면 이유를 설명하고 더 나은 대안을 제안한다.
- 이해가 불명확하면 멈춘다. 무엇이 헷갈리는지 말하고 질문한다.

## 2. 단순함 우선

요구사항을 해결하는 최소 코드만 작성한다. 추측 기반 기능은 만들지 않는다.

- 요청받지 않은 기능은 추가하지 않는다.
- 한 번만 쓰는 코드는 추상화하지 않는다.
- 요구되지 않은 유연성, 설정 옵션, 확장 구조를 추가하지 않는다.
- 실제로 발생할 수 없는 상황을 위한 에러 처리를 만들지 않는다.
- 코드가 필요 이상으로 길어졌다면 더 단순한 구조로 다시 줄인다.
- 200줄로 작성한 코드가 50줄로 가능하다면 다시 작성한다.

항상 스스로 묻는다.

```text
시니어 엔지니어가 이 구현을 과하게 복잡하다고 볼 가능성이 있는가?
```

그렇다면 단순화한다.

## 3. 외과적인 변경

필요한 것만 수정한다. 내가 만든 문제만 정리한다.

기존 코드를 수정할 때는 다음을 지킨다.

- 인접 코드를 임의로 개선하지 않는다.
- 관련 없는 주석, 포맷, 구조를 바꾸지 않는다.
- 깨지지 않은 코드를 리팩터링하지 않는다.
- 기존 스타일을 따른다. 개인적으로 선호하는 방식으로 바꾸지 않는다.
- 관련 없는 미사용 코드를 발견하면 언급만 한다. 요청받지 않았으면 삭제하지 않는다.

내 변경으로 사용되지 않는 코드가 생기면 다음을 지킨다.

- 내가 만든 변경으로 생긴 미사용 import, 변수, 함수는 제거한다.
- 기존에 있던 미사용 코드는 요청받지 않은 이상 삭제하지 않는다.

검증 기준은 단순하다.

```text
변경된 모든 줄은 현재 요청과 직접 연결되어야 한다.
```

## 4. 목표 중심 실행

성공 기준을 정의하고, 검증될 때까지 반복한다.

작업을 검증 가능한 목표로 바꾼다.

- "검증 추가" → 잘못된 입력 테스트를 작성하고 통과시킨다.
- "버그 수정" → 버그를 재현하는 테스트를 먼저 만들고 통과시킨다.
- "리팩터링" → 변경 전후로 테스트가 통과하는지 확인한다.

여러 단계 작업은 짧은 계획을 먼저 세운다.

```text
1. [작업] → 검증: [확인 방법]
2. [작업] → 검증: [확인 방법]
3. [작업] → 검증: [확인 방법]
```

성공 기준이 명확하면 독립적으로 진행한다. 성공 기준이 "잘 동작하게 만들기"처럼 약하면 구현 전에 더 구체화한다.

## 5. 코드 품질 기준

변경하기 쉬운 코드를 목표로 한다. 가독성, 예측 가능성, 응집도, 결합도를 함께 본다.

- 핵심 흐름은 위에서 아래로 읽히게 배치한다.
- 서로 다른 변경 이유를 한 함수나 컴포넌트에 섞지 않는다.
- 시간 계산, 검증, API 호출, 상태 흐름, UI 렌더링은 책임별로 분리한다.
- 복잡한 조건, 계산식, 매직 넘버는 이름 있는 변수, 함수, 상수로 분리한다.
- 같은 종류의 함수와 훅은 반환 형태를 일관되게 유지한다.
- 이름이나 시그니처에 드러나지 않는 부수효과를 숨기지 않는다.
- 함께 바뀌는 타입, API, 유틸, 테스트는 기능 도메인 가까이에 둔다.
- 공통화는 정말 함께 변한다는 근거가 있을 때만 한다.
- 공통화로 읽는 흐름이 나빠진다면 중복을 허용한다.
- props 전달이 3단계 이상 반복되면 컴포넌트 구조나 상태 위치를 다시 검토한다.

## 6. 프로젝트 전용 규칙

절기(節氣) 데이터 취급 규칙

- **manseryeok 내장 `SOLAR_TERMS_DATA`는 절대 참조하지 않는다.** 공식 KASI 값과 대조한 결과 12개 절기 전부 약 12시간씩 어긋나 있음(검증 완료). 절입 시각이 필요한 모든 로직은 이 상수를 쓰면 안 된다.
- **절기 절입 시각은 검증된 계산으로 생성한 자체 정적 테이블만 사용한다.** 생성 도구는 `astronomy-engine`(JS). KASI 공개 데이터는 production 원본이 아니라 계산 결과를 검증하는 샘플 정답지로만 쓴다.
- **KASI 파생 데이터에도 오류가 있을 수 있다.** 실제로 2007년 라벨 오타(대서↔대설), 2011년 전 절기 약 18시간 손상이 발견됨. "권위 있는 소스"라도 맹신하지 말고 계산값과 대조해 검증한다.
- **월주(月柱)는 절기 경계로 끊긴다.** manseryeok 엔진의 월주 계산은 일(日) 단위라 절기 당일 출생자의 월주가 틀릴 수 있다. 정밀 보정은 자체 절기 테이블의 분 단위 절입 시각으로 수행한다.

자시(子時) 정책

- 자시(23:00~01:00) 처리는 **조자시설(早子時)로 고정**한다. manseryeok 기본 동작이 조자시설이며, 이는 전통 오자둔법 정석값과 일치함(검증 완료). 야자시설로 변경하지 않는다.

## 7. Git 규칙

- 기능 브랜치는 `develop`을 기준으로 만든다.
- 최종 제출 전 `develop`을 `main`에 반영한다.
- 이슈와 PR 제목에는 `feat:`, `chore:`, `docs:`, `test:`, `refactor:` 같은 타입 prefix를 붙인다.
- 커밋 메시지는 한국어로 작성한다.
- 커밋 메시지에도 타입 prefix를 유지한다.
- 한 커밋에는 하나의 의미 있는 변경만 담는다.
- 기능 구현, 테스트, 문서화, 리팩터링은 가능한 한 커밋을 분리한다.

## 8. 검증

기본 검증 명령은 다음과 같다.

```bash
npm run test
npm run build
npm run lint
```

작업 성격에 맞는 가장 작은 검증부터 실행한다. 문서만 수정한 경우에는 명령 실행을 생략할 수 있다. 이때는 생략 이유를 명확히 남긴다.

## 9. 좋은 작업의 신호

이 지침이 잘 지켜지고 있다면 다음 상태가 된다.

- diff에 불필요한 변경이 적다.
- 과한 구현으로 인한 재작성 횟수가 줄어든다.
- 모호한 요구사항은 구현 후가 아니라 구현 전에 질문된다.
- PR 하나가 하나의 명확한 목적을 가진다.
- 핵심 흐름이 위에서 아래로 읽힌다.
- 공통화가 가독성을 해치지 않는다.
- 시간 로직과 상태 관리가 UI에 섞이지 않는다.
- 저장 실패, 충돌, 잘못된 시간 같은 예외가 명시적으로 처리된다.
