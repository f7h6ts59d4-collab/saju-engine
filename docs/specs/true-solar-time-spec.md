# 작업 명세: correctPillars에 진태양시(眞太陽時) 보정 추가

> saju-engine 레포. 일주·시주를 출생지 진태양시(경도 + 균시차) 기준으로 계산하도록
> correctPillars를 개정한다. 엔진 내부(src/core)는 건드리지 않고 래퍼만 수정한다.
> 모든 동작은 코드 실험으로 검증 완료(아래 "검증된 사실" 참조).

## 배경 / 문제

현재 correctPillars는 일주·시주를 입력 벽시계 그대로 둔다(주석에 "진태양시는 쓰지
않는다"). 그 결과 출생지 시간대가 일주·시주에 반영되지 않아, 서울/뉴욕/LA의 같은
벽시계 시각이 모두 동일한 일주·시주를 낸다(측정 확인). 미국 타깃 상품에서 이는 오류다.
한국 사주는 미국처럼 경도 폭이 큰 지역은 진태양시 보정이 필요하다.

## 목표

일주·시주를 **출생지 진태양시**(경도 보정 + 균시차 EoT) 기준으로 계산한다.
연주·월주(절기)는 현재 로직(출생 UTC vs 절입 UTC) 그대로 둔다.

## 확인한 사실 (실험으로 확인했으나 2차 검증 권장)

1. **엔진의 기존 경도 보정은 미국에 쓸 수 없다.** src/core/saju.ts의 보정 공식
   `입력분 - (135 - longitude)*4`는 "입력 시각이 한국 표준시(KST)"임을 전제한다.
   미국 출생은 입력이 현지시(예: EDT)라 이 공식에 미국 경도를 넣으면 분 계산이 깨져
   시각이 약 1시간씩 불규칙하게 어긋난다. (뉴욕 1992-07-23을 0~23시로 스캔 시 24개 중
   13개가 올바른 진태양시 결과와 불일치했고, 자정 근처는 일주까지 어긋났다.) 따라서
   미국엔 못 쓰며, 보정을 끄고 HourAngle 방식으로 대체한다.
2. **엔진 보정을 끌 수 있다.** `calculateSaju(..., { applyTimeCorrection: false })`이면
   엔진이 입력 시각을 그대로 쓴다(경도 보정 안 함).
3. **astronomy-engine의 HourAngle로 진태양시를 직접 얻는다.** 이미 의존 중인 라이브러리.
   경도 보정 + 균시차(EoT)가 자동 포함된다(별도 EoT 공식 불필요). 독립 기준과 대조해
   정확성을 2차 확인했다: 천문 EoT 알려진 값(2/11 -14분, 5/14 +4분, 11/3 +16분 등)과
   거의 일치, 경도 1도=4분 일치, 한국 통상 -30분 보정과도 부합(계절 평균 -28분).
4. **엔진의 일주 경계는 자정이다.** getGapja는 연·월·일만 받아 날짜로 일주를 정한다
   (시각 무관). 23:00~23:59=그날, 00:00~00:59=다음날. 이 동작을 유지한다.
5. **날짜 넘김이 처리된다.** 진태양시가 자정을 넘으면(예: KST 00:20 → 진태양시 전날
   23:41) 일주가 전날로 넘어간다(해당 케이스에서 기해 확인).

## 구현 — correctPillars만 수정 (엔진 src/core 불변)

### 입력 시그니처 확장
`BirthInput`에 출생지 좌표 추가:
- `longitude: number` (필수. 도시 대표 경도. 예: 뉴욕 -74)
- `latitude: number` (필수. 도시 대표 위도. 예: 뉴욕 40.7)
  - 진태양시는 경도가 지배적이고 위도 영향은 미미하나, HourAngle이 관측자 위치를
    받으므로 함께 넘긴다. 도시 단위 대표 좌표면 충분(도시 내 오차는 무시).

### 진태양시 계산 (래퍼에 추가)
출생 순간(UTC, 기존 `zonedDateTimeToUtc`로 IANA·DST 자동 반영)과 경도·위도로
진태양시의 **달력 순간**(연·월·일·시·분)을 구한다:

```
1. utc = zonedDateTimeToUtc(...)  // 기존 코드, DST 자동 환원됨
2. observer = new Astronomy.Observer(latitude, longitude, 0)
3. ha = Astronomy.HourAngle(Astronomy.Body.Sun, utc, observer)  // 0~24
4. tstHours = (ha + 12) mod 24                                   // 진태양시 시각
5. offset(시간) = tstHours - utcHourFrac, 단 (-12,12]로 wrap 보정
6. tstInstant = utc + offset시간
7. 진태양시 달력값 = tstInstant의 (UTC기준) Y/M/D/H/M
   → 자정 넘김이 여기서 자동 처리됨(전날이면 day가 자동으로 -1)
```
(검증 코드의 trueSolarParts 로직과 동일하게 구현.)

### 일주·시주 산출
진태양시 달력값으로 엔진을 호출하되 **엔진 보정을 끈다**:
```
const base = calculateSaju(tstYear, tstMonth, tstDay, tstHour, tstMinute,
                           { applyTimeCorrection: false });
```
- base.dayPillar / base.hourPillar 를 일주·시주로 쓴다(진태양시 기준).
- 시간 모름(hour undefined)이면 기존처럼 정오(12:00) 가정 후 진태양시 적용, 시주는 참고용.

### 연주·월주 — 변경 없음 (격리)
- 연주·월주는 **출생 UTC**(진태양시 아님) vs 절입 UTC 비교로 현재 로직 유지.
- 절기는 천문학적 절대 순간이라 UTC 기준이 맞다. 진태양시를 절기 판정에 쓰지 말 것.
- resolveYearPillar / resolveSajuMonth 는 birthUtcMs를 그대로 사용(기존과 동일).

### 주의 (절대 금지)
- src/core/* (엔진 내부) 수정 금지. 래퍼(correct-pillars.ts)와 입력 타입만 수정.
- 엔진의 기존 경도 보정 공식을 미국에 쓰지 말 것(KST 가정이라 깨짐). 반드시 끄고
  (applyTimeCorrection:false) HourAngle 방식으로 대체.
- 연주·월주 판정에 진태양시/경도를 섞지 말 것(UTC 절기 비교 그대로).

## 검증 (테스트 필수)

기존 테스트가 안 깨지는지 + 아래 신규 케이스:
1. **회귀**: 기존 절기·연월주 테스트 전부 통과(연주·월주 로직 불변).
2. **시간대 반영**: 서울/뉴욕/LA 같은 벽시계 → 일주·시주가 **이제 서로 다르게** 나옴
   (기존엔 동일했던 것이 진태양시로 갈림).
3. **날짜 넘김(조자시)**: 서울 1992-07-23 00:20 KST → 진태양시 전날 23:41 → 일주 기해
   (전날). 같은 서울 00:50 → 진태양시 00:11 → 일주 경자(그날). 둘이 갈리는지.
4. **EoT 반영**: 같은 경도·같은 벽시계라도 2월(EoT≈-14분)과 11월(EoT≈+16분)에
   진태양시가 다른지(경계 케이스에서 시주 갈림 확인).
5. **균시차/경도 합산 정확성**: 뉴욕 14:30 EDT → 진태양시 약 13:28 근방인지.
