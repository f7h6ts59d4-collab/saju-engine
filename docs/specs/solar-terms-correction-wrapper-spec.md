# 작업 명세: 절기 경계 월주·연주 보정 래퍼

> Claude Code 작업 지시용 1회성 명세. 구현·검증 완료 후 "완료 기록"으로 보관/정리.
> 권장 위치: `docs/specs/solar-terms-correction-wrapper.md`
> 선행 작업: 절기 빌드 스크립트(`src/correction/solar-terms-precise.json` 생성) 완료되어 있어야 함.

## 목적

**배포 엔진(dist·jest)**은 빌드의 compress 단계가 월주를 절기 기준(`getSajuMonth` + 年上起月法)으로 재계산하므로 이미 절기 기준이다. 다만 (i) **일(日) 단위**라 절기 당일의 분 단위 절입 전/후를 구분하지 못하고(입춘이면 **연주(年柱)**도), (ii) **고정 절기일**을 써서 실제 천문 절입과 ±1일 어긋나는 연·절기가 있다(예: 2024 경칩·청명). 이 래퍼는 자체 정적 절기 테이블(`solar-terms-precise.json`)의 **분 단위 절입 시각**으로 출생 순간(UTC)을 비교해 월주(입춘 시 연주)를 정밀 보정한다.

> 빌드 전 **원본 데이터**(`src/data/date-index.ts`)는 월주가 음력 월초 기준이라 틀리지만, 이는 `tsx`로 src를 직접 실행할 때만 노출되고 배포·테스트 경로에는 쓰이지 않는다 — `docs/issue-5-month-pillar-bug-analysis.md` 참고.

엔진 내부는 수정하지 않는다(엔진 미변경 원칙). 보정은 엔진 출력 위에 얹는 별도 레이어다.

## 핵심 원리 (반드시 이 모델대로 구현)

### 1) 절기 경계는 "전 세계 공통의 천문학적 순간"이다
- 월주/연주가 바뀌는 기준은 절입(節入)이라는 **단일 시점(instant)**이다. 시간대와 무관하게 전 세계 동시에 일어난다.
- 따라서 비교는 **출생 순간(UTC) vs 절입 순간(UTC)**으로 한다. 진태양시(경도·균시차)는 이 보정에 **필요 없다** — 그건 시주(時柱) 보정에만 쓰이며 본 작업 범위 밖이다.

### 2) 월주는 래퍼가 전적으로 소유한다 (年上起月法으로 산출)
- 배포 엔진의 월주는 절기 기준이지만 일 단위·고정 절기일이라, 절기 당일 분 단위 절입과 천문 절입 ±1일 오차에서 어긋난다. 이 둘을 모두 잡으려면 "전날 빌려오기"로는 부족하다(전날 빌려오기는 분 단위만 잡고 ±1일 고정일 오차는 못 잡음).
- 따라서 래퍼는 출생 순간(UTC)이 속한 절기를 precise 테이블에서 판정해 사주월(지지)을 정하고, **연간(年干)에 따른 年上起月法(오호둔법)으로 월주(천간+지지)를 모든 날짜에 대해 산출**한다. 깨끗한 비경계 날에는 결과가 배포 엔진과 일치한다.
- **연주(年柱)**는 배포 엔진 경계가 입춘 '날짜' 기준이라 대부분 올바르고 입춘 당일 분 단위만 어긋난다. 따라서 입춘 절입 전 출생에 한해 `(입춘 당일 − 1일)`을 manseryeok으로 재계산해 그 **연주**를 빌려 전년으로 되돌린다(천간은 엔진이 자동으로 맞춤).
- **일주(日柱)·시주(時柱)는 절대 덮지 않는다.** 실제 출생일·시각 기준 그대로 둔다. 래퍼가 소유하는 것은 월주(+입춘 시 연주)뿐이다.

### 3) 입춘은 특별 처리
- 12절 중 **입춘만** 연주 경계를 겸한다. 입춘 절입 전이면 월주 + 연주 둘 다 전날 값으로 되돌린다.
- 나머지 11절은 월주만 보정한다(연주는 안 바뀜).

## 입력

- 생년월일 (양력/음력 선택)
- 출생 시각 (+ "시간 모름" 옵션)
- 출생 도시 (자동완성 검색, 각 도시에 IANA 시간대 id가 연결되어 있어야 함)
- 성별 등 그 외 입력은 보정 로직과 무관

### 시간대 처리 (서머타임 포함)
- 출생 도시 → IANA 시간대 id(예: `Asia/Manila`)로 매핑.
- 그 시간대 + 출생일시로 **출생 순간을 UTC로 환산**한다. 이때 과거 서머타임(DST)은 IANA 데이터가 자동 반영한다.
- 구현은 런타임 내장 `Intl.DateTimeFormat`(IANA 지원) 우선. 표준 API로 부족하면 가벼운 tz 라이브러리 허용. **별도 대형 데이터셋을 번들에 넣지 말 것** — DST 이력을 직접 표로 만들지 않는다.
- 사용자에게 "UTC offset"이나 "서머타임 여부"를 직접 고르게 하지 않는다. 도시명만 받아 전부 자동 처리한다.

### "시간 모름" 처리
- 시각을 **정오(12:00)로 가정**해 4주를 모두 채운다(정오는 날짜 경계에서 가장 먼 안전 시각).
- 단, 시주는 "출생시간 미상 — 참고용"으로 **표시**한다(틀린 값을 확정처럼 보이지 않게).
- 시간을 몰라도 도시(시간대)는 받는다 — 월주/연주 절기 경계 비교에 여전히 필요.
- 결과에 안내 한 줄: 출생시간을 알면 더 정확하다는 취지(재방문 유도 겸용).
- 주의: 외국 출생 + 시간 모름은 시차로 날짜가 ±하루 흔들릴 수 있다. 이는 라이트 유저 특성상 감수한다(별도 보정 안 함).

## 자시(子時) 정책
- 조자시설(早子時) 고정. manseryeok 기본 동작 유지. 본 래퍼는 시주를 건드리지 않으므로 자시도 엔진 기본값 그대로 둔다.

## 처리 흐름

```
1. 입력(도시·생년월일·시각·양음력) 수집. 음력이면 양력으로 변환.
2. 도시 → IANA tz id → 출생 순간을 UTC로 환산 (DST 자동)
3. manseryeok으로 기본 명식 계산 → 일주·시주만 채택(불변)
4. 월주 (모든 날짜):
   a. solar-terms-precise.json의 12절 절입 시각(KST)을 UTC instant로 환산
   b. 출생 UTC 이하의 가장 최근 절입을 찾아 사주월(지지)을 판정
   c. (보정된) 연간으로 年上起月法(오호둔법) → 월주 산출
5. 연주 (입춘 경계만):
   a. 출생일이 그 해 입춘 "절기 당일"이고 출생 UTC < 입춘 절입 UTC이면
      → (입춘 당일 − 1일)을 manseryeok으로 재계산해 그 연주를 빌려 전년으로 되돌림
   b. 그 외에는 엔진 연주 그대로
6. 시간 모름이면 정오(12:00) 가정, 시주는 "참고용"으로 표시
```

## 필수 검증 단계 (스킵 금지)

검증을 포함하고, 실패 시 빌드/테스트를 실패시킨다.

1. **경계 정확성 테스트**: 절기 절입 시각 직전/직후(예: 절입 −1분 / +1분) 출생을 만들어, 월주(입춘은 연주도)가 정확히 갈리는지 확인. 같은 날 같은 도시에서 절입 전후로 월주가 달라져야 한다(manseryeok 단독은 못 하던 것).
2. **입춘 연주 테스트**: 입춘 절입 전 출생이 연주까지 전년으로 되돌려지는지 확인.
3. **비경계 무변화 테스트**: 분 단위·천문 절입에 모두 부합하는 깨끗한 비경계 출생은 배포 엔진(dist·jest = compressed) 출력과 100% 동일해야 한다(보정이 멀쩡한 값을 건드리지 않음). 단 고정 절기일이 천문 절입과 ±1일 어긋나는 날(예: 2024 경칩·청명)은 래퍼가 더 정확하므로 의도적으로 달라진다.
4. **시간대 테스트**: 동일 UTC 순간을 서로 다른 도시(시간대)로 입력해도 같은 명식이 나오는지 / 시차로 절기 경계를 넘는 케이스가 올바른지 확인.
5. **일주·시주 불변 테스트**: 보정이 일어나도 일주·시주는 안 바뀌는지 확인.

> 테스트 인프라: `npm test`는 jest `moduleNameMapper`가 가리키는 `date-index-compressed.ts`(gitignore, 자동생성)에 의존한다. 이를 위해 `pretest: npm run compress-data`로 테스트 전 compressed 데이터를 생성한다(신규 클론에서 빌드 없이 `npm test` 가능).

## 알려진 함정 (주석으로 명시)

- 진태양시(경도·균시차)는 본 보정에 쓰지 않는다. 월주/연주는 UTC instant 비교로 판정한다.
- **월주 천간은 年上起月法(오호둔법)으로 직접 산출한다.** 배포 엔진 월주는 일 단위·고정 절기일이라 "전날 빌려오기"로는 분 단위만 잡고 ±1일 고정일 오차는 못 잡기 때문이다. (연주만 입춘 경계에서 엔진의 전날 값을 빌린다.)
- 일주·시주를 덮지 말 것. 래퍼가 소유하는 것은 월주(+입춘 시 연주)뿐이다.
- 사용자에게 서머타임/offset을 묻지 말 것. 도시명 → IANA 자동 처리.
- manseryeok 내장 `SOLAR_TERMS_DATA`는 참조 금지(약 12시간 오차). 절입 시각은 `src/correction/solar-terms-precise.json`만 사용.
