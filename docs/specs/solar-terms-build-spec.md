# 작업 명세: 절기 절입 시각 빌드 스크립트

> Claude Code 작업 지시용 1회성 명세. 구현·검증 완료 후에는 "완료 기록"으로 보관하거나 정리.
> 권장 위치: `docs/specs/solar-terms-build.md`

## 목적

`astronomy-engine`(JS)으로 **1900~2050년 12절(節)의 절입 시각**을 계산해, 월주 보정 래퍼가 참조할 정적 데이터 파일을 생성한다. 빌드 시점에 한 번만 실행하는 개발 스크립트이며, 런타임에는 생성된 JSON만 사용한다(네트워크·변동비 0).

## 범위

- **12절만** 생성한다(월주 경계용). 중기(中氣) 12개는 월주와 무관하므로 제외.
- 대상 연도: **1900 ~ 2050**.

## 의존성

- `astronomy-engine` (npm, JS 버전) — **devDependency로 설치**. 런타임 번들에 포함하지 않는다.

## 계산 로직

각 절기를 태양 겉보기황경(도)에 매핑:

| 절기 | 황경 | 절기 | 황경 |
|---|---|---|---|
| 소한 | 285 | 소서 | 105 |
| 입춘 | 315 | 입추 | 135 |
| 경칩 | 345 | 백로 | 165 |
| 청명 | 15  | 한로 | 195 |
| 입하 | 45  | 입동 | 225 |
| 망종 | 75  | 대설 | 255 |

- 각 연도·각 절기마다 `SearchSunLongitude(목표황경, 시작시각, 40)`로 도달 시각(UTC)을 구한다.
- 시작시각은 각 절기의 대략적 시작 월 1일로 잡는다(소한 1월, 입춘 2월 … 대설 12월).
- **UTC 결과를 반드시 KST(UTC+9)로 변환**해 저장한다. KASI 절입 시각이 E135=KST 기준이므로, 이 변환을 해야 정답지와 맞는다. (Python 검증에서 이 변환으로 평균 ~12초 오차까지 일치 확인됨.)

## 출력 형식

분 단위까지 저장(초 단위 불필요 — KASI도 분 단위). 연도를 키로 하는 구조:

```json
{
  "2024": [
    { "name": "소한", "lon": 285, "kst": "2024-01-06T05:49" },
    { "name": "입춘", "lon": 315, "kst": "2024-02-04T17:27" }
  ]
}
```

절기 배열은 황경 순이 아니라 양력 시간 순(소한→입춘→…→대설)으로 정렬한다.

## 필수 검증 단계 (스킵 금지)

스크립트는 생성만 하고 끝내면 안 된다. **자체 검증을 포함**하고, 검증 실패 시 빌드를 실패(exit code ≠ 0)시킨다.

1. 동봉된 KASI 정답지 파일(`kasi_solar_terms_truth_2004_2026.json`)을 로드한다.
2. 정답지의 각 연도·절기를, 같은 계산 로직으로 구한 값과 **분 단위로 대조**한다.
3. **2011년은 제외**한다(정답지 `_meta.known_bad_years` 참조 — distbe 원본이 약 18시간 손상됨).
4. 합격 기준: 2011 제외 전 항목이 **1분 이내로 일치**해야 한다(Python 검증 시 264/264 통과, 평균 오차 ~12초).
5. 하나라도 1분을 초과하면 빌드 실패 + 어긋난 항목(연도·절기·계산값·정답값·차이[분])을 출력한다.
6. 검증 통과 후에만 1900~2050 전 범위 파일을 기록한다.

## 알려진 함정 (주석으로 코드에 명시할 것)

- **manseryeok 내장 `SOLAR_TERMS_DATA`를 참조하지 말 것.** 공식 KASI와 약 12시간 어긋남. 검증·생성 어디에도 쓰지 않는다.
- **KASI 정답지 자체에도 오류가 있음.** 2007 라벨 오타(교정 완료), 2011 약 18시간 손상(제외 대상). 그래서 production 데이터는 KASI 복사가 아니라 계산으로 생성하고, KASI는 검증 샘플로만 쓴다.
- UTC→KST 변환 누락이 가장 흔한 실수. 변환을 빠뜨리면 9시간 어긋나 검증에서 전부 실패한다.

## 산출물

1. `scripts/build-solar-terms.{ts,mjs}` — 생성 + 자체 검증 스크립트
2. 생성된 정적 데이터 파일 — **보정 전용 위치 `src/correction/solar-terms-precise.json`**에 저장.
3. `package.json`에 빌드 스크립트 등록(예: `"build:solar-terms"`)

### 출력 위치 관련 필수 제약

- 생성 데이터는 **`src/data/`에 두지 않는다.** `src/data/solar-terms-data.ts`(틀린 `SOLAR_TERMS_DATA`의 출처)와 섞이면 "엔진은 옛 데이터, 래퍼는 새 데이터"의 혼동이 생긴다. 보정 전용 폴더 `src/correction/`에 분리한다.
- **기존 `src/data/` 파일을 수정·덮어쓰지 않는다.** manseryeok 엔진은 그대로 두고, 월주 보정은 래퍼가 이 새 파일을 참조해 수행한다(엔진 미변경 원칙).
- **기존 `scripts/`의 변환기(`convert-solar-terms.ts`, `convert-to-js-data.ts`, `dump-mysql-data.ts`, `compress-date-index.ts`)는 수정하지 않는다.** 이들은 manseryeok 원본 데이터 파이프라인(MySQL/backend 의존)으로 본 작업과 목적이 다르다. 참고만 한다.
